# B109 F002 — Tushare 单次调用静默截断（实测审计）

**日期：** 2026-07-20
**发现者：** Generator（B109 F002 实施过程中）
**严重度：** ★★ 高 — 使 **F001 已交付报告中的核心数字作废**
**状态：** 代码已修（分页 + 触顶守卫）；F001 重测**待用户裁定**

---

## 1. 结论先说

**Tushare 的单次 API 调用会静默截断返回行数——不报错、不抛异常、不置任何标志位。**
调用方拿到的是一个「看起来很像全量」的整数行数。

更关键的是：**截断不是均匀采样，它富集了 vintage 记录**——即被砍掉的那部分，
恰恰是修订率、`flag=0` 保留率这些 PIT 指标赖以计算的行。

因此 **B109 F001 报告 `docs/audits/B109-F001-vintage-probe-2026-07-20.md` 中
所有基于 `income_vip` 的数字都是在残缺数据上算出来的**，且偏差方向是**系统性低估修订率**。

---

## 2. 实测证据

### 2.1 两个接口，两个不同的上限

| 接口 | 单次返回 | 分页拉全 | 漏掉 | 漏掉占比 |
|---|---|---|---|---|
| `income_vip`（2022FY） | **9,000** 行 | 10,093 行 | 1,093 行 | **10.8%** |
| `namechange`（无参） | **10,000** 行 | 11,414 行 | 1,414 行 | **12.4%** |

复现命令：

```bash
.venv/bin/python -c "
import tushare as ts
from scripts.research.ashare_pit.vintage_probe import load_token
pro = ts.pro_api(load_token())
print(len(pro.income_vip(period='20221231', fields='ts_code,f_ann_date,update_flag,n_income_attr_p')))
"   # → 9000（恰好停在整数上，这是唯一的线索）
```

`income_vip` 2022FY 单次 5,673 只证券 → 分页 5,984 只，**311 只证券整个消失**。

`namechange` 漏掉的 1,414 条**全部是最早期记录**（`000001.SZ` 'S深发展A' 20061009 等）——
即历史截面最需要的那一段。

### 2.2 ★截断非均匀：富集 vintage 记录

2022FY 的 `update_flag` 分布：

| | 单次 | 分页全量 | 漏掉 |
|---|---|---|---|
| `update_flag=0`（首次披露版） | 3,419 | 4,206 | **18.7%** |
| `update_flag=1`（修正版） | 5,581 | 5,887 | 5.2% |

`flag=0` 行的漏失率是 `flag=1` 的 **3.6 倍**。而 `flag=0`/`flag=1` 配对正是 F001
用来检出修订的主要机制——截断直接削弱了探针的检出能力。

### 2.3 对 F001 核心数字的影响（分页重测）

| 期间 | 单次行数 | 分页行数 | `flag=0` 保留率（单次→分页） | 观测修订率（单次→分页） |
|---|---|---|---|---|
| **2021FY** | 9,000 | 10,740 | 69.4% → **93.1%** | 0.525% → **1.325%（2.5 倍）** |
| **2022FY** | 9,000 | 10,093 | 60.1% → **70.2%** | 1.534% → **1.721%** |
| 2023FY | 6,728 | 6,728 | 10.5% → 10.5% | 1.032%（未变） |
| 2022H1 | 5,426 | 5,426 | 4.0% → 4.0% | 0.019%（未变） |

**未触顶的期次数字完全不变**，这反过来证实了偏差源就是截断本身。

## 3. 对 F001 已交付结论的逐条影响

| F001 结论 | 影响 | 说明 |
|---|---|---|
| 修订率区间 **0.47%–0.88%** | ❌ **作废** | 系统性低估。仅重测的 4 期里就有 3 期超出该区间上界（1.325% / 1.721% / 1.032%） |
| `flag=0` 保留率 **10.5%–95.4% 剧烈波动** | ⚠️ **部分为伪影** | 2021FY 69.4%→93.1%。波动幅度被截断放大，「保留率机制在迁移」这一叙事需重新验证 |
| 「检出机制从 0/1 配对迁移到多条 flag=1」 | ⚠️ **需重测** | 该判断基于各期 flag 组成，而组成正被非均匀截断扭曲 |
| as-of 可重建性 **98.8%** | ⚠️ **需重测** | 分母是「已知修订数」，而已知修订数本身被低估 |
| **轻量两字段 resolver 成立** | ✅ **未被推翻** | 该裁定取决于「可重建性」与 `f_ann_date` 语义，不取决于修订率高低 |
| 「修订风险集中在 FY 分量」 | ✅ **反而更强** | 重测后 FY 三期均 >1%，2022H1 仅 0.019% |

**净影响：** F002 的**设计**（轻量 resolver + `daily_basic` 分母）不需要改；
但 F001 交付的**测量数字**需要在分页下重跑，且「修订稀有」这一印象需要上调——
重测的 FY 期次修订率在 **1.0%–1.7%**，而非原报告的 0.47%–0.88%。

| `pipeline.py` | **撤回**硬编码的 `REVISION_RATE_BOUNDS = (0.0047, 0.0088)` 与逐报告类型率值，置 `None`；改为只列已分页重测的 4 期；新增 `upstream_fetch_defect` 披露字段随每份报告输出 |

**留 `None` 而不是留旧值**是有意的：一个已知偏低的数比留空更危险，
下游会把它当作已验证的事实引用。

回归测试：`tests/unit/test_ashare_pit_fetch.py`（13 例），
其中 `test_pagination_keeps_going_past_a_full_page` 与
`test_single_fetch_flags_a_capped_looking_result` 直接复现本缺陷形态
（假接口按 `server_cap` 模拟服务端静默砍断）。

## 6. 方法论沉淀（建议入 `framework/proposed-learnings.md`）

**「供应商返回一个整数行数」本身就是需要证伪的假设。**

本缺陷之所以能存活到 F002，是因为 F001 的探针**从未验证过自己拿到的是全量**——
9,000 这个数字看起来完全合理（A 股约 5,000 只股票 × 约 1.8 版本/只）。
没有任何错误信号，只有一个恰好是整数的行数。

与 B108 的 E01 教训（「被规则挡住不等于被验证过」）同类：
**「调用成功返回」不等于「拿到了全部」。** 凡是分页接口，
要么分页拉取，要么显式证明未触顶——不存在第三种可接受状态。
