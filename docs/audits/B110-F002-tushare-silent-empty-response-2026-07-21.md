# B110 F002 实测 — Tushare **静默空响应**（新失败模式，比 B109 的静默截断更危险）

**日期：** 2026-07-21
**发现于：** B110 F002 面板 CLI 的真实数据 smoke 跑
**性质：** 数据源缺陷实测记录，research-only

---

## 1. 现象

`income_vip` / `stock_basic` 在压力或限流下会**返回空表而不抛异常**。

| 端点 | 参数 | 第一次 | 随后连续 4 次 |
|---|---|---|---|
| `income_vip` | `period=20191231, report_type=2` | **0 行** | 4,856 行（稳定） |
| `income_vip` | `period=20200630, report_type=2` | **0 行** | 5,000 行（稳定） |
| `stock_basic` | `list_status=L` | **0 行**（落盘为空文件） | 5,529 行 |
| `income_vip` | `period=20120630 / 20160930 / 20200331`（rt=1） | **0 行** | 正常 |

同一次 smoke 跑中，另有若干调用走的是**抛异常**路径（`fetch.py` 的 3 次重试耗尽后
记入 `failures`）。**两种失败模式并存**，而只有后者会被现有机制发现。

---

## 2. ★为什么它比抛异常危险得多

`fetch.py::fetch_paged` 的终止条件是**短页**（`len(page) < page_size`）。
**空的第一页与「这一期本来就没有数据」在结构上完全无法区分**：

- `report.failures` → 空
- `report.truncation_suspected` → `False`（`looks_truncated(0)` 为假，0 不在 `KNOWN_ROW_CAPS`）
- 覆盖漏斗 → 一切正常闭合
- `marketcap.identity_error` → 依旧 100% 通过（它只校验行内自洽）

**整整一期的数据凭空消失，而所有仪表都是绿的。**

### 2.1 落在 `income_vip` rt=1 上的后果

该期所有证券的分量都会被记成 `COMPONENT_MISSING` → 表现为
「**数据源的固有覆盖限制**」。这正是最难发现的一类污染：它不像 bug，像事实。

且它会**非随机地**打击特定年份（哪一期恰好碰上限流），直接放大覆盖污染项——
2014 年 8.7% 的覆盖缺口对应约 0.87%/年 的污染，几乎等于整条 NO-GO 线（1.0%）。

### 2.2 ★★落在 `stock_basic(list_status='L')` 上的后果（本次实测命中）

**宇宙退化为只剩 338 只退市名**（D 态），而：

- 漏斗照常闭合（分母也小了）
- `in_universe_delisted_later` 探针失效（此时**全部**都是退市名，看起来「含退市名」做得极好）
- 身份校验、恒等式、单调性检查全部照常运行并给出「合理」的数字

这是**幸存者偏差的镜像形态**：不是漏掉退市名，而是只剩退市名。
本次 smoke 跑的缓存里确实落下了一个空的 `stock_basic_L.csv.gz`，
若无本文 §3 的行数下限，它会被后续每一次重跑当作事实复用。

### 2.3 对 B109 已签收代码的影响

`scripts/research/ashare_pit/panel_cli.py:99` 的 `fetch_universe()` 用
`fetch_single_checked` 拉 `stock_basic`，**既无分页也无行数下限**。它对本失败模式
**完全无防护**：空响应 → `rows=0` → `truncation_suspected=False` → 无 failure。

> B109 signoff 的宇宙层结论（幸存者偏差 8.91% 已消除）在**那一次运行**中成立，
> 但该代码路径**不具备可重复性保证**。建议 Planner 排期修复；B110 不改该文件
> （保全 B109 signoff 证据），改为在 B110 自己的抓取层实现下述防线。

## 3. 防线（B110 F002 已实施）

`scripts/research/ashare_pit/ep_panel_cli.py::_fetch_cached`：

```python
_LONG_BACKOFF_SECONDS = (15.0, 30.0, 60.0, 120.0)

for backoff in (0.0, *_LONG_BACKOFF_SECONDS):
    frame, report = fetch_paged(...)
    if report.failures:            continue      # 抛异常路径
    if len(frame) < min_rows:      continue      # ★静默空表路径
    _store(...); return frame                    # 只有通过下限才落盘
raise RuntimeError(...)                          # 取值源：中止，不用残缺数据继续
```

三条缺一不可：

1. **每个端点一个行数下限**（不是「非空」——短表同样可疑）：

   | 端点 | 下限 | 依据 |
   |---|---|---|
   | `income_vip` rt=1 | 1,500 | A 股 2011 年起在市 ≥2,200 只 |
   | `income_vip` rt=2 | 1,000 | 诊断源，耗尽后**如实披露为不可用**而非中止 |
   | `stock_basic` L | 3,000 | 实测 5,529 |
   | `trade_cal` | 3,000 | 实测 4,930 |
   | `daily_basic` / `adj_factor` | 1,500 | 实测 2,400–5,400 |

2. **长退避重试**（15/30/60/120 秒）。`fetch.py` 内建的 3/6 秒退避不足以穿过分钟级限流。

3. ★**绝不缓存空表或短表**。一旦落盘，这个洞在后续每一次重跑里都会变成「事实」，
   而缓存正是让 F003/F004 零成本迭代的机制——它同时也是让错误固化的机制。

**严重度分级**：取值源（rt=1 / 宇宙 / 价格）低于下限 → **中止**，拒绝用残缺数据继续；
诊断源（rt=2 对拍）→ 记 `CROSSCHECK_UNAVAILABLE` **如实披露**，不中止。
★「无法检验」必须与「已检验通过」分开计数——这是 B109 审计器自欺陷阱的同型防护。

## 4. 与 B109 F002 静默截断的关系

| | B109 F002（截断） | B110 F002（空响应） |
|---|---|---|
| 现象 | 单次调用恰好返回 9,000 / 10,000 行 | 返回 **0** 行 |
| 可检出性 | `looks_truncated` 命中已知 cap 时**可能**报警 | **完全不可检出**（0 不在 cap 表，短页即终止） |
| 偏差性质 | 非均匀（被砍掉的行富集 `update_flag=0`） | 整期/整表消失 |
| 防线 | 分页 + 触顶守卫 | **行数下限 + 长退避 + 不缓存空表** |

两者共同的教训：**「拉到的就是全量」是一个断言，不是一个事实**。
B109 已经把它变成可复核的（`FetchReport` 留痕），B110 补上了它在另一个方向上的漏洞。
