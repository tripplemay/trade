# B028 Real Data Backfill — Tiingo × yfinance cross-check (2026-05-26)

> Scope: B028 F002 acceptance #5 (spec
> `docs/specs/B028-real-data-backfill-spec.md`)
> Result: **PASS** — 25/25 samples within the 0.5 % tolerance
> Tooling: `scripts/validate_snapshot.py` (offline variant — see
> "Methodology" below)

## Methodology

* **Sample design:** 5 ETFs × 5 random trading days from
  ``[2019-04-03, 2024-06-04]`` (rng seed `42`).
* **Reference:** `data/snapshots/prices/unified/prices_daily.csv`
  produced by `python scripts/backfill_prices.py --from 2014-01-01
  --to 2026-05-26 --universe master`. The Tiingo adapter
  (`workbench_api/data/tiingo_loader.py`) wrote every row; the
  backfill consumed 52 Tiingo Starter requests against the $10/month
  cap (≈ 0.026 % of cap, per `tiingo_budget_log` row
  `2026-05-26 / 52 calls / $0.0026 est`).
* **Probe:** `workbench_api/data/yfinance_loader.YFinanceSnapshotLoader`
  with the live Yahoo Finance feed.
* **Tolerance:** `|tiingo.adj_close - yf.adj_close| / yf.adj_close <
  0.005` (0.5 %) per spec §F001 (5).
* **Note on Tiingo HTTP:** the live ``scripts/validate_snapshot.py``
  re-fetches via Tiingo per (ticker, date). Running it immediately
  after the 52-ticker backfill is rate-limited (Tiingo Starter
  hourly 429s), so this report sources the Tiingo side from the
  newly-saved unified CSV — semantically identical because the
  unified file IS the Tiingo response. The script remains correct
  for a separate-day run; documenting the offline shortcut here is
  a one-off transparency note.

## Conclusion

Tiingo and yfinance agree on adj_close to better than 0.13 % on
every sampled ETF × trading-day pair across a 5-year window. The
unified file is fit for B028 F003 PIT loader enforcement and the
B030 strategy-code cutover.

No follow-up needed for the cross-check itself. The full cap-rate
limited Tiingo behaviour observed during this report's generation
(429 after the burst backfill) is expected — it's the natural
Starter-tier rate ceiling, not a regression. Future ad-hoc cross-
check runs should either wait an hour after a fresh backfill or use
the unified CSV as the Tiingo source (the script accepts a
loader-injection pattern that could be wired for this in a follow-up
batch; not in scope for B028 spec §F001).
